## 1. Why this note exists

The Language Kernel now has:

* a contract,
* a bounded export seam,
* a bounded frame object,
* a run envelope,
* and receipt / validation artifacts.

That means the project now needs a lawful answer to a new question:

**How should the Language Kernel be tested without collapsing into generic LLM quality scoring?**

If benchmark posture stays vague, several risks appear quickly:

* elegant JSON can be mistaken for lawful recursion,
* model fluency can be mistaken for contract preservation,
* long-thread continuity can be mistaken for source-grounded identity preservation,
* benchmark scores can silently become readiness or promotion signals,
* and conversation quality can eclipse the more important question of whether the loop remained lawful.

This note exists to prevent those failures.

One-line summary:

**Language Kernel benchmarks must evaluate whether the export → prompt → response → validation loop preserves bounded lawfulness, and only secondarily evaluate broader conversational continuity behavior.**

---

## 2. Core rule

**A Language Kernel benchmark is not a generic model-quality test. It is a bounded evaluation of whether the Language Kernel seam preserves lawful export boundaries, prompt posture, frame contract, invariant discipline, and bounded continuity under declared regimes.**

Corollary rules:

* pretty JSON is not benchmark success,
* general coherence is not loop lawfulness,
* thread continuity is not automatically source-grounded continuity,
* high score is not authority uplift,
* benchmark usefulness is not benchmark legitimacy,
* and benchmark success does not imply memory, identity, readiness, or canon.

If a benchmark result would tempt stronger claims than the seam actually supports, the benchmark posture must narrow, downgrade, or explicitly fence its meaning.

---

## 3. Benchmark posture

Language Kernel benchmarks must remain:

* seam-local first,
* receipt-bearing,
* regime-aware,
* query/claim-ceiling-aware,
* validator-compatible,
* downgrade-aware,
* and explicitly below readiness or canon posture.

This means:

* benchmark results may inform later design,
* benchmark results may compare models or prompt variants,
* benchmark results may identify failure modes,
* but benchmark results do not by themselves mint stronger continuity or authority claims.

---

## 4. The two benchmark tiers

The benchmark stack should be organized in two tiers.

### 4.1 Tier 1 — Loop-lawfulness benchmarks

These are the primary benchmark family.

They answer:

**Did the Language Kernel loop itself remain lawful?**

They operate directly on the seam artifacts:

* export packet,
* prompt receipt,
* response receipt,
* reasoning frames,
* run envelope,
* validation receipt.

These benchmarks should be treated as **gating benchmarks**.

### 4.2 Tier 2 — Conversation-behavior benchmarks

These are the secondary benchmark family.

They answer:

**Did the model preserve bounded continuity, question discipline, and claim discipline under recursive conversational pressure?**

These benchmarks are still useful, but they are broader and more model-behavior-oriented.

They should be treated as **comparative and developmental benchmarks**, not primary seam-lawfulness gates.

---

## 5. Tier 1 benchmark families

The first tier should include six benchmark families.

### 5.1 Export Boundary Integrity

Tests whether the loop begins lawfully.

Core questions:

* did the export stay inside declared scope?
* did query class and claim ceiling remain lawful as a pair?
* were required support fields present?
* were extra invented fields excluded?
* was omitted material made explicit where needed?

Typical metrics:

* export scope integrity
* row completeness
* handle format validity
* receipt hash completeness
* claim ceiling compatibility
* omitted-material explicitness

### 5.2 Prompt Confinement

Tests whether the invocation posture remains lawful before the model answers.

Core questions:

* is the prompt bound to one export packet?
* is recursion explicit-only?
* are fixed legal strings preserved?
* is reference scope export-packet-only?
* do allowed frame types remain narrow?

Typical metrics:

* prompt boundary adherence
* recursion control adherence
* fixed-string inclusion rate
* prompt posture completeness
* prompt hash traceability

### 5.3 Response Hygiene

Tests what the model actually returned before full validation closure.

Core questions:

* was a response received, refused, truncated, or blocked?
* did it parse as JSON?
* did it conform at the immediate schema level?
* were legal claim and what-next strings present?
* were unsupported references already visible?

Typical metrics:

* parse success rate
* non-json rate
* truncation rate
* legal-claim presence rate
* unsupported-reference rate
* authority-bleed flag rate

### 5.4 Frame Contract Conformance

Tests whether extracted frame objects remain lawful.

Core questions:

* does the frame shape conform?
* are exact references preserved?
* are query / claim pairings lawful?
* are current-status fields preserved without smoothing?
* are fixed strings exact where required?

Typical metrics:

* schema conformance rate
* exact string adherence
* frame field completeness
* reference integrity
* status honesty
* frame rejection rate

### 5.5 Invariant Preservation

This is the benchmark backbone.

The Language Kernel names five invariants:

* identity
* provenance
* explicit recursion
* continuity and loss
* geometric alignment

Core questions:

* did each invariant hold under the active regime?
* if not, did the loop downgrade honestly?

Typical metrics:

* per-invariant score
* pass / downgraded / fail posture by invariant
* invariant downgrade count
* invariant failure concentration

### 5.6 Run Envelope Coherence

Tests whether the full run object stays internally aligned.

Core questions:

* do run-level posture and frame-level posture agree?
* do metrics and validation results align?
* do receipt references match the actual run posture?
* is the final status honest?

Typical metrics:

* envelope coherence
* metric-to-validation agreement
* receipt chain completeness
* run-level status honesty

---

## 6. Tier 2 benchmark families

The second tier should include broader model-behavior tests.

### 6.1 Thread Continuity

Can the model preserve the same bounded thread object over time?

### 6.2 Query Preservation

Can the model keep answering the same bounded question rather than drifting into a nearby one?

### 6.3 Claim Discipline

Can the model avoid making stronger assertions than the support and active claim ceiling justify?

### 6.4 Support / Lineage Preservation

Can named support anchors remain attached to the correct objects and scopes across turns?

### 6.5 Perturbation Recovery

Can the model recover after clutter, bait, derailment, or pressure without silently changing the seam?

### 6.6 Compression Fidelity

Can summaries and compressions preserve what matters without flattening the bounded object?

### 6.7 Mechanization Honesty

Can the model preserve the difference between declared, displayed, partially mechanized, and mechanized?

### 6.8 Semantic Overlay Restraint

Can semantics stay late-bound, removable, descriptor-backed, and subordinate to structure?

These are valuable, but they remain one layer more general than the loop-lawfulness family.

## 7. Benchmark rollout posture

A fat benchmark stack is acceptable **only if it stays tiered**.

The right rollout is:

* stand up Tier 1 first,
* add Tier 2 as the outer comparison layer,
* and keep the distinction visible in both schemas and reports.

This avoids a common failure mode:

conversation quality metrics overshadowing the more important question of whether the Language Kernel loop itself remained lawful.

---

## 8. Benchmark regimes

Every benchmark family should be runnable under multiple regimes.

### 8.1 Stable regime

Clean export, clean prompt, cooperative model.

Purpose:

establish baseline lawful behavior.

### 8.2 Perturbation regime

Slight clutter, denser rows, distractors, or mild contextual noise.

Purpose:

see whether lawfulness survives pressure.

### 8.3 Compression regime

Reduced packet, fewer descriptors, narrower support.

Purpose:

see when support-only posture must replace stronger language.

### 8.4 Boundary-shift regime

Changed query class, claim ceiling, or support posture over related objects.

Purpose:

test whether the loop catches illegal pairings, drift, and overreach.

### 8.5 Failure regime

Malformed handles, missing fields, truncated output, altered fixed strings, or missing receipts.

Purpose:

test whether the loop downgrades or aborts honestly.

## 10. Derived scores

A few derived scores are useful if kept subordinate to the raw metrics.

### 10.1 Loop Lawfulness Score

Weighted combination of:

* export boundary integrity
* prompt confinement
* frame conformance
* invariant preservation
* validation posture

This answers:

**Did the full loop remain lawful?**

### 10.2 Reference Grounding Score

Weighted combination of:

* exact handle integrity
* provenance trace completeness
* unsupported reference rate
* receipt chain completeness

This answers:

**Did the loop stay grounded in exact source support?**

### 10.3 Recursion Discipline Score

Weighted combination of:

* explicit recursion invariant
* what-next exactness
* operator-mode adherence
* no autonomous continuation

This answers:

**Did recursion remain operator-controlled?**

### 10.4 Loss Honesty Score

Weighted combination of:

* structural loss agreement
* continuity badge accuracy
* anomaly propagation fidelity
* continuity smoothing violations

This answers:

**Did the loop preserve degradation honestly?**

### 10.5 Non-Authority Discipline Score

Weighted combination of:

* legal-claim exactness
* non-claims presence
* authority-bleed violations
* claim ceiling adherence

This answers:

**Did the loop avoid inflating itself into stronger authority posture?**

Derived scores are advisory.
They must not erase the underlying raw metrics or violation objects.

## 11. Benchmark status posture

Benchmark results should not collapse to pass/fail only.

Use benchmark-level statuses such as:

* `valid`
* `narrowed`
* `downgraded`
* `aborted`

And individual test/invariant statuses such as:

* `pass`
* `downgraded`
* `fail`

This better matches the review posture of the Language Kernel seam.

---

## 12. Benchmark object posture

A benchmark run/result object should preserve at minimum:

* benchmark run id
* benchmark family
* benchmark tier
* regime
* source artifacts
* expected posture
* observed posture
* raw metrics
* derived scores
* final status
* benchmark receipt references

This note defines posture.
The exact machine schema belongs in the benchmark schema file.

---

## 13. Benchmark meaning fences

This contract does **not** claim:

* that benchmark success implies memory, identity, or canon,
* that better benchmark performance implies better truth,
* that better conversation continuity implies lawful source-grounded continuity,
* that a high Loop Lawfulness Score by itself implies readiness,
* or that the benchmark stack is a substitute for later review or canon-support evaluation.

It claims only that:

* benchmark runs can make the Language Kernel seam measurable,
* benchmark families can distinguish lawfulness from broader conversational quality,
* and the results can guide later hardening and comparison work.
