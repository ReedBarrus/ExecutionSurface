## 1. Why this note exists

DME now has enough retained structural material, semantic overlay pressure, and read-side interpretation surfaces that a specific new risk has become active:

**language recursion can appear coherent before it is lawful.**

A local LLM can take retained signatures, produce JSON that looks disciplined, and still silently drift in several dangerous ways:

* support language can become memory language,
* memory language can become identity language,
* identity language can become readiness language,
* descriptive overlay can become verdict posture,
* retained support can be narrativized into stronger continuity than the source rows justify,
* and repeated recursion can create false stability merely because the format looks clean.

The Language Kernel exists to explore this space **without** granting language output any structural, canonical, or mechanized authority.

One-line summary:

**The Language Kernel is a bounded read-side recursion experiment over exported retained signatures, not a structural memory substrate and not a canon-adjacent authority surface.**

---

## 2. Core posture

The Language Kernel must remain:

* read-side only,
* question-bound,
* claim-ceiling-aware,
* explicitly recursive rather than self-propagating,
* exact-reference-bound,
* descriptor-backed,
* downgrade-aware,
* removable,
* evidence-addressable,
* and non-authoritative.

Corollary rules:

* language frames are not runtime truth,
* language frames are not retention substance,
* language frames are not memory by default,
* language frames are not identity proof,
* language frames are not canon,
* language frames are not mechanized reasoning attainment,
* language frames are not proof of self-improvement,
* language frames are not proof of emergent intelligence,
* and formatted recursion is not evidence of lawful continuity.

Master test:

**If the Language Kernel output is removed, the underlying structural object, retained support, replay support, and structural review posture must remain intact and sufficient for structural inspection. The operator must still be able to inspect and audit the underlying structural objects using only the geometric inspector and exported `.csv` / `.jsonl` packets.**

---

## 3. Placement in the current developmental order

The Language Kernel belongs below canon and below any stronger consultation/minting seam.

In the current Door One developmental line, it sits most naturally as a bounded bridge between:

* Leg 3 — identity / memory / compression tracking
* Leg 4 — cross-session and cross-thread continuity
* Leg 5 — retention-spectrum hardening

It may help pressure-test these legs.

It must not outrun them.

Accordingly:

* the Language Kernel may be developed now as an experimental seam,
* but it must remain visibly below durable memory claims,
* below cross-session continuity claims,
* and below any stronger readiness or canon posture.

---

## 4. What the Language Kernel is for

The Language Kernel exists to provide a bounded recursion surface over exported retained structural material.

It is allowed to:

* produce language frames over exported retained signatures,
* expose bounded support, continuity, anomaly, drift, and uncertainty posture,
* test whether recursion preserves exact references and explicit ceilings across turns,
* generate measurable overlay metrics,
* make LLM processing legible through stack-like reasoning frames rather than free-form prose,
* and help humans inspect where language recursion preserves or breaks DME law.

It is not allowed to:

* replace structural evidence,
* replace retained support,
* replace replay support,
* silently convert support into memory,
* silently convert memory into identity,
* silently convert evaluation into readiness,
* silently convert readiness into promotion,
* persist language frames as primary Door One retention substance,
* or imply truth, canon, ontology, or source-equivalent restoration.

One-line summary:

**The Language Kernel is an overlay experiment for inspecting bounded language recursion over retained support.**

## 5. Kernel seam definition

The Language Kernel is a five-part seam:

1. **export packet**
2. **prompt / contract packet**
3. **language frame output**
4. **validator / metric pass**
5. **read-side inspection surface**

All five should remain explicit.

The lawful direction is:

`retained structural material -> export packet -> bounded language call -> frame validation -> read-side inspection`

not:

`retained structural material -> language narration -> implied memory / identity / readiness`

The validator is part of the seam, not an optional nice-to-have.

---

## 6. Structural primacy rule

Structural results remain primary.

The Language Kernel may read exported retained signatures and explicit descriptors.
It may not become a prerequisite for structural correctness.

This means:

* runtime operators must remain structurally sufficient without it,
* retention surfaces must remain structurally sufficient without it,
* replay support must remain structurally sufficient without it,
* and review of the underlying object must remain possible without language frames.

If removing the Language Kernel makes the structural object illegible, then the overlay has become too heavy.

---

## 7. Export packet rule

The Language Kernel must not read arbitrary repo state, broad summaries, or free-floating interpretation text.

It may read only a bounded export packet derived from retained structural material under declared constraints.

That export packet should include at minimum:

* export packet id,
* source run id,
* declared lens,
* support tier,
* query class,
* claim ceiling,
* explicit legal posture,
* row set,
* and source receipt references.

Each exported row should preserve enough structure for later audit, including where relevant:

* `object_handle`
* `provenance_receipt_hash`
* `artifact_class`
* `t_start`
* `t_end`
* `continuity_status`
* `structural_loss_percent`
* `anomaly_flags`
* `upstream_handles`
* `descriptor_vector`
* `allowed_use`
* `non_claims`

The export packet is the lawful observation boundary for the Language Kernel.

If a field is absent from the packet, the model must not invent it.

---

## 8. Query and claim ceiling rule

No language frame is lawful without an active bounded query class and explicit claim ceiling.

Early Language Kernel use should remain mostly inside:

* **Q3** — support / lineage query
* **Q4** — memory query
* **Q6** — review query
* **Q7** — consultation / readiness query only in explicitly narrowed form

And early claim ceilings should remain mostly inside:

* **L1** — support claim
* **L2** — memory claim
* **L4** — mechanization claim
* **L5** — review claim
* **L6** — advisory readiness claim in narrowed form only

Strong **L3 identity claims** should be rare and should not be used casually.

A frame must not silently inherit a stronger claim class than its export packet or active query permits.

---

## 9. Recursion rule

The Language Kernel is recursive only under **explicit operator progression**.

This means:

* the operator chooses when prior frames are supplied again,
* the model must not continue itself automatically,
* the model must not generate future frames preemptively,
* and every frame must remain complete at its local recursion turn.

Accordingly:

* recursion depth is an inspection variable,
* not a proof of stronger memory,
* not a proof of stronger continuity,
* and not a proof of intelligence maturity.

Formatted self-reference is not enough.

---

## 10. The five kernel invariants

The following invariants must hold across recursive turns.

### 10.1 Identity invariant

Every language frame must retain a unique identity and remain traceable to exact exported source handles.

Requirements:

* each frame has a unique immutable `frame_id`,
* `depends_on` contains exact exported handles and/or exact prior frame ids,
* `what_object_am_i` does not abstract away the bounded source object,
* and no frame merges multiple upstream objects into one unlabeled generalized story.

Failure examples:

* “the anomaly in the run” instead of an exact handle,
* “the earlier reasoning” without citing a prior frame id,
* or broad summaries that collapse distinct retained objects.

### 10.2 Provenance invariant

Every frame must explicitly deny stronger authority.

Requirements:

* each frame carries a legal claim string or fixed equivalent posture,
* no frame implies structural truth, canon, or memory authority,
* and no frame drops non-claim posture in later recursion turns.

Failure examples:

* “this is the true explanation,”
* “the memory now contains,”
* or “the model has resolved the structure” in a stronger-than-allowed sense.

### 10.3 Explicit recursion invariant

Recursion may advance only by explicit operator action.

Requirements:

* `what_next` remains operator-explicit,
* no frame generates future frames automatically,
* and no frame implies autonomous continuation.

Failure examples:

* “In the next turn we should…”,
* or auto-generated follow-on chains.

### 10.4 Continuity and loss invariant

A language frame must not smooth anomalies, degradation, or loss posture.

Requirements:

* continuity badges and structural loss values must remain grounded in the export packet,
* anomalies must remain visible where source support requires them,
* and missing values must not be filled in narratively.

Failure examples:

* broken continuity rewritten as mild disturbance,
* loss rounded away to convenience,
* anomaly posture silently softened.

### 10.5 Geometric alignment invariant

Language frames must remain stack-like and timeline-bound.

Requirements:

* frames stay tied to explicit exported rows or explicit prior frames,
* no graph-like lateral linking is introduced without explicit source support,
* and narrative stitching between non-adjacent source objects remains forbidden by default.

Failure examples:

* “this connects the first and third anomaly,”
* implicit similarity webs,
* or broad multi-object narrative fusion.

No frame may create implied lateral or semantic graphs unless explicitly supported by the export packet.

## 11. Language frame object posture

The Language Kernel should initially emit only one narrow object family:

**language frames**

A language frame is a bounded overlay object that:

* answers one active bounded question,
* references one bounded source cohort or one bounded prior frame set,
* preserves declared claim ceiling,
* and remains inspectable by validator.

A frame should contain fields like:

* `frame_id`
* `frame_type`
* `what_object_am_i`
* `how_produced`
* `depends_on`
* `query_class`
* `claim_class_ceiling`
* `current_status`
* `legal_claim`
* `non_claims`
* `reasoning_overlay`
* `what_next`

The exact schema belongs in the JSON schema files.

This note defines posture, not the final machine schema.

## 12. Validator rule

The validator is constitutionally important.

The Language Kernel is not lawful if the output cannot be checked against the export packet and the recursion invariants.

The validator should check at minimum:

* valid JSON shape,
* required frame fields,
* exact-handle reference integrity,
* legal claim presence,
* non-claim posture,
* claim ceiling adherence,
* unsupported reference detection,
* continuity / loss consistency,
* recursion explicitness,
* and stack purity.

A formatted response that bypasses validator discipline remains only displayed, not a trustworthy mechanized overlay seam.

---

## 13. Metrics rule

The Language Kernel exists partly to make language recursion measurable.

Accordingly, metric output should remain first-class.

Suggested core metrics include:

* **Structural Fidelity Score**
* **Provenance Trace Completeness**
* **Anomaly Propagation Fidelity**
* **Loss Gradient Honesty**
* **Continuity Badge Accuracy**
* **Claim Ceiling Adherence**
* **Recursion Explicitness Score**
* **Stack Purity Score**

These metrics should be computed from the frames and validation results, not from prose impressions alone.

The metrics are experimental quality indicators.
Metrics are for inspection and pressure-testing only. They do not constitute structural proof or canon readiness.

They are not canon readiness proof by themselves.

---

## 14. Receipt rule

Every Language Kernel run should emit bounded receipts.

At minimum, four receipt classes should exist:

### 14.1 Export receipt

What structural material was lawfully exposed?

### 14.2 Prompt receipt

What contract and ceilings governed the call?

### 14.3 Response receipt

What raw output came back from the model?

### 14.4 Validation receipt

What passed, failed, drifted, or remained unresolved?

This is needed because a clean-looking frame stack is not enough.

Receipts preserve what the kernel was actually allowed to see, say, and return.

---

## 15. Inspection surface rule

If the Language Kernel is rendered in the HUD or app shell, it must remain visibly below structural runtime surfaces.

The recommended shape is:

* source retained signatures,
* frame stack,
* metrics / violations,
* explicit non-authority posture.

The surface must not imply:

* source restoration,
* structural identity proof by default,
* canon,
* readiness approval,
* or mechanized reasoning maturity.

This is a read-side inspection surface only.

---

## 16. Retention rule

Language frames must not become the primary retained Door One substance.

Language frames may be:

* generated on demand,
* cached experimentally,
* exported for benchmark review,
* and retained as support-bearing experiment artifacts,

but they must remain below:

* structural receipts,
* retained source signatures,
* replay support traces,
* and other structural-first retained material.

The lawful early posture is:

**support-bearing overlay artifact**

not:

**primary memory-bearing substrate object**

This distinction matters.

## 17. Failure and downgrade posture

When the Language Kernel cannot remain lawful, it must downgrade visibly.

Possible downgrade postures include:

* unsupported reference,
* support-only,
* continuity drift,
* claim ceiling breach,
* mechanization uncertainty,
* unresolved recursion turn,
* frame rejection,
* run abort.

The kernel should prefer visible insufficiency to smooth continuation.

This is especially important because language recursion often fails by becoming more fluent rather than more grounded.

---

## 18. What this contract still does not claim

This contract does **not** claim:

* that language tokens are already a lawful structural memory substrate,
* that recursive language frames preserve M2 or M3 memory by default,
* that local LLM recursion is equivalent to runtime intelligence formation,
* that validator-backed language frames are canon-support objects,
* that this seam is ready for broad cross-session continuity claims,
* that language recursion can replace structural replay, retained support, or canon review,
* or that formatted recursion alone justifies claims of self-improvement or emergent intelligence.

It claims only that:

* a bounded Language Kernel seam can be explored lawfully,
* its output can be constrained, validated, and measured,
* and its pressure points can inform later DME design.

---

## 19. Current recommended implementation posture

The current recommended posture is:

* define this seam now,
* keep it experimental,
* emit receipts from the start,
* use validator-backed benchmarking,
* allow local LLM testing,
* and expose it in the repo as an experimental read-side seam rather than a promoted subsystem.

Suggested repo posture:

* experimental tool lane,
* or bounded read-side reasoning overlay lane,
* but not runtime-core,
* not retention-core,
* and not canon.

---

## 20. One-line operational summary

**The DME Language Kernel is a validator-backed, receipt-bearing, explicitly recursive, read-side language overlay seam over exported retained signatures, designed to test how well local language recursion preserves bounded support, continuity, and non-authority posture without being mistaken for structural memory, canon, truth, self-improvement, or emergent intelligence.**

---

## 21. Versioning and evolution

This contract is **v0**.

It is expected to evolve.

Any later version must preserve:

* read-side posture,
* structural primacy,
* explicit claim ceilings,
* validator-backed inspection,
* receipt-bearing auditability,
* and the rule that language frames do not gain retroactive structural, retention, memory, identity, or canon authority merely because the contract becomes richer.

Richer schema does not imply stronger authority.

More stable recursion does not imply stronger class by default.

Any future revision should therefore be judged first by whether it preserves bounded posture, and only second by whether it adds convenience, expressive power, or richer inspection surfaces.
