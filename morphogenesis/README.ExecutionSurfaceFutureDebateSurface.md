# ExecutionSurface Future Debate Surface

## Status

This document collects future-facing ideas worth preserving after the
architecture rebase, without allowing them to outrank the current runtime
grammar corpus, live architecture seed, live subject registry, or live
implementation ladder.

It is a holding surface for later translation and debate.

It is not current runtime law.

It is not current workflow law.

## Purpose

Several older morphogenesis support files still contained useful pressure, but
their framing was tied to the older recursion stack, older object trunk, or
older cleanup phases.

This surface preserves the worthwhile ideas in a clean, non-authoritative form
so they can be reopened later without dragging their original containers
forward.

## Debate families

The extracted material currently falls into four debate families:

1. workflow and proof grammar
2. runtime and exposure guardrails
3. cleanup and tightening residue
4. field posture and external anchor disciplines

## 1. Workflow and proof grammar

These ideas are worth preserving for the later workflow reseed.

### A. Evaluation / memory split

Future workflow law should likely preserve this split:

- `Evaluate` audits, reviews, reconciles, and decides immediate outcome
- `Memorize` retains, archives, reconstructs, promotes, or otherwise routes
  memory posture

Reconstruct and promote should not silently collapse into evaluation just
because they are convenient.

### B. Topology-first workflow grammar

When workflow is reseeded, it should likely center these proof objects:

1. `Topology detection object`
2. `Identity object`
3. `Decision object`
4. `Evaluation object`
5. `Memory record or archive entry`

This is now preferable to the older recursion packet stack because it forces the
workflow to name what topology it is protecting before it starts smoothing.

### C. Coherence inflation guardrail

Future workflow law should likely preserve a version of this question:

Did the pass reduce ambiguity lawfully, or merely smooth it into something that
looks coherent enough to release?

The useful parts to preserve are:

- ambiguity-hiding smoothing should not release a subject
- authority-inflating resolution should lose write privilege or decision force
- elegant prose should not override active topological or object-law tension

### D. Stabilizing-pass legitimacy

Between major subjects and implementation rungs, stabilizing passes should
remain lawful.

Their job is to:

- catch hidden inflation
- tighten boundaries
- harden non-claims
- reduce scope bleed
- improve replayable clarity

They should not silently open new architecture layers.

### E. Deferred-upper-subject discipline

Even in the reseeded workflow, it may remain useful to keep later branches
visibly pinned so lower-layer work does not silently compensate for them too
early.

## 2. Runtime and exposure guardrails

These items remain important, but are now secondary to the live runtime grammar
corpus rather than separate doctrine.

### A. Runtime grammar inheritance

Any future workflow, audit, or implementation pass that touches runtime-facing
object law should explicitly inherit the runtime grammar corpus.

That includes at minimum:

- `README.ExecutionSurfaceRuntimeGrammarCorpus.md`
- `README.ExecutionSurfaceRuntimeObjectGrammar.md`
- `README.ExecutionSurfaceRuntimeEmissionAuthorityGrammar.md`

And when seam placement or topology is in scope:

- `README.ExecutionSurfaceRuntimeChainMap.md`
- `README.ExecutionSurfaceRuntimeSeamLedger.md`
- `README.ExecutionSurfaceRuntimeObjectTopologyAudit.md`

### B. Exposure guardrail

The older structural/support/receipt exposure rule still contains a useful
warning:

no meaningful runtime seam from support recruitment upward should degrade into
receipt-only visibility.

That idea is now stronger in the live corpus:

- direct support-object exposure is mandatory where honest carriage remains
  practical
- receipts are subordinate
- descriptors and object cards are fallback views only

### C. Commit mini-gate residue

A future workflow or validator layer may still want a small admission or commit
pre-check.

Useful candidate checks are:

- required object class or object family named
- required provenance anchors present
- required policy anchors present
- authority posture declared
- rejection conditions named

The older requirement for a rejection receipt shape should not be treated as
current law, but may be useful later as a validator or audit helper.

## 3. Cleanup and tightening residue

The old tightening checklist still contains useful cleanup pressure.

These items remain worth revisiting:

### A. Documentation residue sweep

- descriptor-first language
- receipt-first language
- any old retained-object trunk assumptions

### B. Compression seam review

- `runtime/lm/WorkbenchLmWrapper.js`
- `runtime/CrossRunDynamicsReport.js`
- live provenance and digest scripts

These are still the most likely places for early flattening or transport
overreach.

### C. Benchmark and runner hygiene

- compact and diffable benchmark outputs
- explicit and boring LM invocation path
- minimal intentional fixture set

These are practical cleanup targets, not architecture law.

## 4. Field posture and external anchor disciplines

The old field-posture surface still carries useful long-horizon framing.

### A. Structural memory architecture

The phrase is still useful:

`ExecutionSurface` may be treated as work in structural memory architecture.

That phrase remains useful so long as it does not outrank the concrete runtime
grammar.

### B. Adjacent fields worth drawing from

Still-useful anchor disciplines include:

- signal processing
- dynamical systems
- information geometry
- machine learning
- neuroscience

These may contribute:

- mathematics
- measurement methods
- modeling tools
- stability criteria
- divergence measures
- reconstruction limits

They should not silently decide:

- substrate truth
- commit-boundary law
- object identity
- canon or truth posture
- authority posture

### C. Anti-dissolution posture

The useful warning to preserve is:

borrowed methods are lawful; borrowed authority is not automatic.

## What this surface is for

This surface is the right place to park:

- future workflow reseed ideas
- unresolved proof-grammar questions
- future visualization or projection questions
- cross-discipline framing worth revisiting
- cleanup residue that is real but not current trunk authority

## What this surface is not for

This surface must not become:

- a second architecture seed
- a second runtime grammar corpus
- a shadow workflow law
- a dumping ground for active implementation requirements

## One-line summary

This future-debate surface preserves worthwhile workflow, guardrail, cleanup,
and field-posture ideas from older morphogenesis support files so they can be
translated later without letting those older files remain active authority.
